# ADR-008: Qibla API Integration with Caching and Fallback

**الحالة:** Accepted  
**التاريخ:** 31 يناير 2026  
**المؤلفون:** Mobile Architecture Lead

---

## السياق

التطبيق يحتاج إلى دقة عالية في حساب اتجاه القبلة. حالياً، يستخدم التطبيق حساب محلي فقط (`QiblaCalculator`). لتوفير دقة أعلى وموثوقية أفضل، نريد:

1. **تكامل مع API خارجي** (aladhan.com) للحصول على اتجاه القبلة المحسوب بدقة عالية
2. **استراتيجية تخزين مؤقت** لتقليل استهلاك البيانات وتحسين الأداء
3. **استراتيجية fallback** لضمان عمل التطبيق حتى عند فشل API

## القرار

نقرر تصميم `QiblaService` مع:

### 1. API Integration
- إضافة `fetchQiblaDirection` في `APIClient` للتكامل مع `https://api.aladhan.com/v1/qibla/{latitude}/{longitude}`
- استخدام async/await مع timeout قصير (5 ثوانٍ)
- معالجة الأخطاء بشكل مناسب

### 2. Caching Strategy
- **مدة الصلاحية:** 24 ساعة
- **مفتاح الكاش:** يعتمد على الإحداثيات مع تقريب (precision: 0.01 درجة ≈ 1.1 كم)
- **البنية:** JSON يحتوي على `direction`, `latitude`, `longitude`, `timestamp`
- **التحقق:** التحقق من صلاحية الكاش وعمر البيانات قبل الاستخدام

### 3. Fallback Strategy
- **الترتيب:**
  1. التحقق من الكاش أولاً
  2. محاولة جلب من API مع timeout
  3. استخدام الحساب المحلي (`calculateLocalBearing`) كـ fallback
- **الموثوقية:** يضمن عمل التطبيق دائماً حتى بدون اتصال بالإنترنت

### 4. Dependency Injection
- `QiblaService` يستقبل `APIClient` و `LocalCache` عبر constructor
- يتبع نفس النمط المستخدم في `PrayerTimeService` و `AzkarService`

## البدائل المدروسة

### البديل 1: استخدام API فقط بدون fallback
**السلبيات:**
- التطبيق لا يعمل بدون اتصال بالإنترنت
- استهلاك بيانات أعلى
- اعتماد كامل على خدمة خارجية

**الإيجابيات:**
- دقة عالية دائماً من API

**القرار:** ❌ مرفوض - الموثوقية أهم من الدقة الإضافية

### البديل 2: استخدام الحساب المحلي فقط
**السلبيات:**
- لا نستفيد من دقة API المحسنة
- قد تكون هناك اختلافات طفيفة في الحساب

**الإيجابيات:**
- لا يحتاج اتصال بالإنترنت
- بسيط وسريع

**القرار:** ❌ مرفوض - نريد دقة أعلى عند توفر API

### البديل 3: Caching بدون expiration
**السلبيات:**
- قد تكون البيانات قديمة
- لا يتم تحديث الاتجاه عند تغيير الموقع بشكل كبير

**الإيجابيات:**
- استهلاك بيانات أقل

**القرار:** ❌ مرفوض - نحتاج تحديث دوري للبيانات

### البديل 4: Caching مع expiration + API + Local Fallback (المقترح)
**السلبيات:**
- تعقيد أكبر قليلاً في الكود

**الإيجابيات:**
- ✅ دقة عالية من API عند توفرها
- ✅ عمل بدون إنترنت عبر fallback
- ✅ تقليل استهلاك البيانات عبر caching
- ✅ موثوقية عالية

**القرار:** ✅ مقبول - أفضل توازن بين الدقة والموثوقية والأداء

## العواقب

### الإيجابية
- ✅ دقة أعلى في اتجاه القبلة عند توفر API
- ✅ عمل التطبيق حتى بدون اتصال بالإنترنت
- ✅ تقليل استهلاك البيانات عبر caching
- ✅ تحسين تجربة المستخدم (سرعة الاستجابة)
- ✅ يتبع نفس النمط المعماري المستخدم في الخدمات الأخرى

### السلبية
- ⚠️ تعقيد أكبر قليلاً في `QiblaService`
- ⚠️ يحتاج تحديث `AppContainer` لتمرير dependencies
- ⚠️ قد يحتاج تحديث الكود الذي يستخدم `QiblaService.bearing()` لاستخدام async/await

### ملاحظات التنفيذ
- `bearing()` أصبح async function - يحتاج تحديث الكود المستخدم
- الكاش يستخدم precision 0.01 درجة لتجميع المواقع القريبة
- Timeout 5 ثوانٍ للـ API calls لتجنب الانتظار الطويل
