# ADR-007: Adaptive Update Rate Manager

**الحالة:** Accepted  
**التاريخ:** 30 يناير 2026  
**المؤلفون:** Knowledge Engineer

---

## السياق

البوصلة تحتاج تحديثات مستمرة، لكن:
1. **عند الثبات**: لا نحتاج تحديثات عالية (5 Hz كافية)
2. **عند الحركة**: نحتاج تحديثات عالية (30 Hz للدقة)
3. **استهلاك البطارية**: التحديثات العالية تستهلك بطارية أكثر

الحل الثابت (معدل تحديث ثابت) إما:
- عالي جداً (يستهلك بطارية) عند الثبات
- منخفض جداً (دقة أقل) عند الحركة

## القرار

نقرر تطبيق **Adaptive Update Rate Manager** لتعديل معدل التحديث تلقائياً بناءً على حالة الحركة.

### المميزات الرئيسية:
- **تكيف تلقائي**: تعديل معدل التحديث بناءً على الحركة
- **توفير البطارية**: معدل منخفض عند الثبات
- **دقة عالية**: معدل عالي عند الحركة
- **Throttling ذكي**: منع التحديثات غير الضرورية

## البدائل المدروسة

### البديل 1: معدل تحديث ثابت عالي (30 Hz)
**السلبيات:**
- استهلاك بطارية عالي عند الثبات
- تحديثات غير ضرورية
- حرارة الجهاز

**الإيجابيات:**
- دقة عالية دائماً
- بسيط

**القرار:** رُفض - استهلاك بطارية غير مقبول

### البديل 2: معدل تحديث ثابت منخفض (5 Hz)
**السلبيات:**
- دقة أقل عند الحركة
- استجابة بطيئة
- تجربة مستخدم سيئة

**الإيجابيات:**
- توفير بطارية
- بسيط

**القرار:** رُفض - دقة غير كافية

### البديل 3: Adaptive Update Rate ✅
**السلبيات:**
- أكثر تعقيداً
- يحتاج كشف الحركة

**الإيجابيات:**
- توازن مثالي بين الدقة والبطارية
- تكيف تلقائي
- تجربة مستخدم جيدة

**القرار:** ✅ **مقبول** - أفضل حل للتوازن

### البديل 4: يدوي (المستخدم يختار)
**السلبيات:**
- يحتاج تدخل المستخدم
- تجربة مستخدم سيئة
- معظم المستخدمين لا يعرفون ما يختارون

**الإيجابيات:**
- مرونة للمستخدم
- بسيط من ناحية التطبيق

**القرار:** رُفض - غير عملي

## العواقب

### الإيجابية
- ✅ **توفير البطارية**: تقليل الاستهلاك بنسبة 50-70% عند الثبات
- ✅ **دقة عالية**: معدل عالي عند الحاجة
- ✅ **تكيف تلقائي**: لا يحتاج تدخل المستخدم
- ✅ **تجربة مستخدم**: استجابة سريعة عند الحركة
- ✅ **Throttling ذكي**: منع التحديثات غير الضرورية

### السلبية
- ⚠️ **تعقيد**: خوارزمية كشف الحركة
- ⚠️ **تأخير في التكيف**: يحتاج وقت للكشف عن التغيير
- ⚠️ **استهلاك موارد**: حساب السرعة الزاوية

### المحايدة
- 📊 **معلومات إضافية**: توفير معلومات عن حالة الحركة
- 🔧 **قابل للتخصيص**: العتبات قابلة للتعديل

## التنفيذ

### حالات الحركة:
```swift
enum MotionState {
    case stationary      // ثابت - 5 Hz
    case slowMovement    // حركة بطيئة - 15 Hz
    case fastMovement    // حركة سريعة - 30 Hz
}
```

### الخوارزمية:
```
angularVelocity = calculateAngularVelocity(headingHistory)
if angularVelocity < stationaryThreshold:
    state = .stationary (5 Hz)
else if angularVelocity < slowMovementThreshold:
    state = .slowMovement (15 Hz)
else:
    state = .fastMovement (30 Hz)
```

### المعاملات:
```swift
stationaryThreshold: 0.5 deg/s    // عتبة الثبات
slowMovementThreshold: 5.0 deg/s   // عتبة الحركة البطيئة
historyWindow: 0.5 seconds        // نافذة التحليل
```

## الملاحظات

- العتبات قابلة للتعديل بناءً على نتائج الاختبارات
- يمكن إضافة تحسينات مثل Performance-aware throttling
- النافذة الزمنية (0.5 ثانية) تعطي توازن جيد بين الاستجابة والاستقرار

## التحسينات المستقبلية

- Performance-aware throttling: تقليل المعدل عند ارتفاع CPU
- Machine Learning: استخدام ML للتنبؤ بالحركة
- User preferences: إمكانية اختيار المستخدم للمعدل المفضل
