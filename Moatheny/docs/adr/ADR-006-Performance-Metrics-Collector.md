# ADR-006: Performance Metrics Collector

**الحالة:** Accepted  
**التاريخ:** 30 يناير 2026  
**المؤلفون:** Knowledge Engineer

---

## السياق

البوصلة المحسنة تستخدم خوارزميات معقدة (EKF، Sensor Fusion) قد تؤثر على الأداء. نحتاج:
1. **مراقبة الأداء**: تتبع CPU, Memory, Latency
2. **كشف المشاكل**: تحديد bottlenecks
3. **ضبط الأداء**: تحسين بناءً على القياسات
4. **ضمان الجودة**: التأكد من الالتزام بـ Performance Budgets

بدون مراقبة الأداء، قد نواجه:
- استهلاك بطارية عالي
- تأخير في الاستجابة
- تجربة مستخدم سيئة

## القرار

نقرر تطبيق **Performance Metrics Collector** لمراقبة وقياس أداء البوصلة بشكل مستمر.

### المميزات الرئيسية:
- **قياسات شاملة**: CPU, Memory, Latency, Update Rate
- **Performance Budgets**: حدود واضحة للأداء
- **تقارير**: تقارير مفصلة عن الأداء
- **تحذيرات**: تحذيرات عند تجاوز الحدود

## البدائل المدروسة

### البديل 1: بدون مراقبة الأداء
**السلبيات:**
- لا نعرف إذا كان هناك مشاكل أداء
- صعوبة في تحديد bottlenecks
- قد نكتشف المشاكل بعد فوات الأوان

**الإيجابيات:**
- بسيط
- لا استهلاك موارد إضافي

**القرار:** رُفض - غير مقبول للتطبيق المحسن

### البديل 2: مراقبة يدوية (Debug فقط)
**السلبيات:**
- لا يعمل في Production
- يحتاج تدخل يدوي
- غير مستمر

**الإيجابيات:**
- بسيط
- لا استهلاك موارد في Production

**القرار:** رُفض - نحتاج مراقبة مستمرة

### البديل 3: Performance Metrics Collector ✅
**السلبيات:**
- استهلاك موارد إضافي (ضئيل)
- يحتاج كود إضافي

**الإيجابيات:**
- مراقبة مستمرة
- تقارير مفصلة
- كشف المشاكل مبكراً
- قابل للتفعيل/تعطيل

**القرار:** ✅ **مقبول** - أفضل حل للمراقبة المستمرة

### البديل 4: استخدام أدوات خارجية فقط (Instruments)
**السلبيات:**
- يحتاج Xcode
- غير متاح في Production
- لا يعطي معلومات في الوقت الفعلي

**الإيجابيات:**
- أدوات قوية
- تحليل مفصل

**القرار:** رُفض - يُستخدم كأداة مساعدة، ليس بديلاً

## العواقب

### الإيجابية
- ✅ **شفافية**: معرفة أداء النظام في الوقت الفعلي
- ✅ **كشف مبكر**: اكتشاف المشاكل قبل أن تؤثر على المستخدم
- ✅ **ضبط الأداء**: تحسين بناءً على القياسات
- ✅ **ضمان الجودة**: التأكد من الالتزام بـ Performance Budgets
- ✅ **قابل للتفعيل/تعطيل**: يمكن تعطيله في Production

### السلبية
- ⚠️ **استهلاك موارد**: استهلاك CPU و Memory إضافي (ضئيل)
- ⚠️ **كود إضافي**: يحتاج كود للقياس والتسجيل
- ⚠️ **صيانة**: يحتاج تحديث عند تغيير الكود

### المحايدة
- 📊 **معلومات إضافية**: توفير معلومات عن الأداء للمطورين
- 🔧 **قابل للتخصيص**: Budgets قابلة للتعديل

## التنفيذ

### القياسات:
- **CPU Usage**: نسبة استخدام المعالج
- **Memory Usage**: استخدام الذاكرة (bytes)
- **Latency**: وقت المعالجة (ms)
- **Update Rate**: معدل التحديث الفعلي (Hz)
- **Filter Processing Time**: وقت معالجة EKF (ms)

### Performance Budgets:
```swift
CPU Usage: < 5% average
Latency: < 16ms (60fps)
Memory: < 10MB for compass processing
Filter Processing: < 1ms
```

## الملاحظات

- يمكن تعطيل Performance Monitoring في Production لتقليل الاستهلاك
- القياسات تعطي تقديرات (ليست دقيقة 100%)
- يمكن إضافة تحسينات مثل Export للتحليل الخارجي
- Budgets قابلة للتعديل بناءً على نتائج الاختبارات
